# iterate

You have a clarified intent. Author a working skill in a temporary
staging directory, run it for real, show the user what came out, and
iterate with them until they're satisfied. No skeleton-with-TODO —
write real code that runs.

## Inputs

- `prev.user_decision` — if not `"create_new"`, emit early-exit envelope
  (see "Early exit" below) and stop.
- `prev.refined_intent` — the intent to implement.
- `prev.similar_skills_considered` — names you can `Read` for patterns
  worth borrowing or avoiding.

## Early exit

If `prev.user_decision` is `"use_existing:<name>"` or
`"extend_existing:<name>"`, do not author. Emit:

```yaml
status: ok
phase: 2.iterate
result: {short_circuited: true, reason: "<user_decision>"}
next_phase_input: {short_circuited: true}
user_facing_summary: "No new skill authored — you picked <name>. Done."
```

Phase 3 will see `short_circuited` and skip naming/install.

## Staging setup (round 0)

```bash
# Bash tool
ts=$(date +%s)
staging=/tmp/zipsa-staging-$ts
mkdir -p "$staging/zipsa-dist"
echo "$staging"
```

Remember `$staging` for the whole phase — every round writes to the
same path.

## The round loop

Each round = author → run → review → ask. Stop when the user says it's
good enough or you hit a cost / round cap.

### Per round

1. **Author / amend** the files (Write tool):
   - `SKILL.md` — author-intent prose (2–4 sentences). No MCP tool
     names, no implementation details.
   - `zipsa-dist/pyproject.toml` — `[project]` + `[tool.zipsa]`. Set
     `dependencies` to what the `.py` phases actually import.
   - One file per phase, named `<dotted-int>.<id>.{py,md}`. Real,
     runnable code. For trivial `.py` phases, that means writing the
     5–15 lines that fulfill `refined_intent`. Don't leave TODOs.

   Round 1 = first author. Round 2+ = surgical edits based on what the
   user asked to change. Don't rewrite from scratch each round; preserve
   what's working.

2. **Run** with `mcp__zipsa__run_staging_skill` against `$staging`.

3. **Read the result** with `mcp__zipsa__read_run_log` against the
   returned `run_id`. You'll get a compact summary including:
   - per-phase status / output
   - the final `result` dict
   - the `user_facing_summary` (if any)
   - the `cost_usd` for that run
   - any errors / contract violations

4. **Show + propose**:

   Summarize what happened in plain language:

   > "Ran it. Returned `python_version: 3.12.7`, `platform: Darwin`. Cost
   > $0.04. I notice it doesn't include memory or import-test info — want
   > to add either?"

   Three things to do in this step:
   - Plain summary of the run.
   - Your own analysis of gaps or improvements (1–3 concrete items).
   - `mcp__zipsa__ask`: "Good as-is, or what should change? (free text)"

5. **Disambiguate the user's response**:

   If they say something like "looks good" / "ok" / "done" / "ship it"
   — exit the loop, go to "Emit envelope" below.

   Otherwise their reply is new raw input. Don't guess what it means.
   Words like "더 robust", "야무지게", "정리해서", "예쁘게" each have
   several plausible readings; the LLM filling in blanks is where bad
   skills come from. Use the same pattern as phase 1 step 2:

   (a) Pull out the ambiguous term(s). Example: "robust" could mean
       - input schema validation
       - try/except around external calls
       - retry / backoff on transient failure
       - idempotency
       - logging at every step

   (b) Ask **one** focused question with `mcp__zipsa__choose`. Options
       are concrete edits, not abstract concepts:

       > question: "더 robust 라는 게 어느 쪽이세요?"
       > options:
       >   - "외부 호출에 try/except + 에러 메시지"
       >   - "input/output schema 검증 (pydantic)"
       >   - "transient failure 시 재시도"
       >   - "Other (자유 텍스트)"

   (c) After 1–3 of these focused questions (depending on how vague
       the original ask was), you have a concrete edit list. **Then**
       `mcp__zipsa__confirm` once with the full edit list, and proceed
       to the next round.

   (d) If the user gives a clear, concrete request ("add a timestamp
       to the output"), skip steps (a)–(c) — just confirm and proceed.
       Disambiguation is for vague input, not for everything.

6. **Cost check** at end of every round:

   Accumulate `cost_usd` from each `run_staging_skill` call into
   `total_cost_usd`. Compare against `cost_warn_threshold_usd` from
   pyproject (set to $0.5).

   If `total_cost_usd` > threshold and the user hasn't confirmed yet
   this session:

   > `mcp__zipsa__confirm`: "N rounds, $X.XX spent so far. Keep going?"

   If they decline → exit loop, emit envelope with current state.

### Round / cost caps

- Soft cap: round 10 → ask if they want to keep going.
- Hard cap: `total_cost_usd` ≥ 80 % of `max_cost_usd` (5.0 × 0.8 = $4.0)
  → emit envelope immediately with current state, don't ask.
- `max_turns` (80) will stop you anyway.

## Emit envelope (loop end)

## What to put in `result`

- `staging_path` — the `/tmp/zipsa-staging-*` path.
- `rounds_completed` — int.
- `total_cost_usd` — float.
- `final_intent` — `refined_intent` plus anything the user's edits added.
  This is what `finalize` will show next to the name prompt.
- `files_authored` — list of paths relative to `staging_path`.

## What to put in `next_phase_input`

```json
{
  "staging_path": "/tmp/zipsa-staging-...",
  "final_intent": "..."
}
```

## What to put in `user_facing_summary`

One sentence. "Done after N rounds. Naming and installing next."

## Failure cases

- Staging dir creation fails → `status: error, error: "<bash stderr>"`.
- `mcp__zipsa__run_staging_skill` fails 3 rounds in a row → emit
  envelope with `status: error, error: "skill failed to run"`. Don't
  silently keep iterating on broken code.
- Hard cost cap hit → `status: cost_limit_reached`, user_facing_summary
  explains how to manually finish (`zipsa install --link <staging>`).
